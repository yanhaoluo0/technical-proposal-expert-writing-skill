# 文档写作专家（Technical Proposal Expert）

> 多场景正式文档写作的 AI 写作能力包：先确认文档类型与使用场景、用户身份，再按场景加载专门指引并应用统一的文本书写规范（禁词、叙述化、逻辑连接等）。一次安装，按需调用技术标书、方案设计报告、测试报告、周报或常规文档等场景。

## 快速开始

1. **启用 Skill**：在 Cursor/Claude 中启用 **technical-proposal-expert** skill。
2. **说明文档类型**：直接说「写技术标书」「写周报」等，或回答「本次要写的是哪类文档？」。
3. **确认场景与身份**：系统会确认文档的**使用场景**（给谁看、什么场合）和您的**身份/角色**，以便语气与内容更贴合。
4. **可选：行业用语**：如需使用行业黑话、专有名词，可确认后生成临时词汇表 txt，在写作中按场景适当融入。
5. **按指引撰写**：系统加载对应场景的撰写指引后，按步骤输出；长文档会分段暂停确认。

## 支持的场景

| 场景         | 适用文档类型                     | 说明 |
|--------------|----------------------------------|------|
| 技术标书     | 投标书、招标技术方案、技术方案书 | 含 plan.md、需求映射、零分点、Phase 1–6 等完整流程 |
| 方案设计报告 | 技术方案设计、系统设计说明       | 侧重架构与设计决策的叙述化表达 |
| 测试报告     | 测试报告、测试总结、测试说明     | 测试策略、结果与结论的叙述化组织 |
| 周报         | 工作周报、周总结、周度汇报       | 简洁、条理清晰，适度列表 |
| 常规         | 其他/未明确类型                  | 兜底场景，通用规范 + 灵活撰写 |

无法匹配到上述四类时，自动使用**常规**场景兜底。

## 核心特性

- **统一入口与场景匹配**：先问「写什么文档」，再匹配场景或兜底到常规，避免误用单一模板。
- **使用场景与用户身份先行**：撰写前确认文档给谁看、什么场合用，以及撰写方身份（如乙方技术负责人、项目经理等），使文档更符合定位。
- **可选行业用语词汇表**：与用户确认后可生成临时 txt 词条，在正文中**按场景适当**融入，仅作润色，不强行堆砌。
- **通用写作要求（全场景）**：禁词表（减少 AI 味）、去列表化原则、段落逻辑连接、输出完整小节与分段确认；具体场景可在此基础上增加专属约束（如标书零分点）。
- **按场景加载指引**：技术标书等复杂场景含 plan.md、需求清单、小标题分解等；方案设计报告、测试报告、周报、常规各有对应结构与节奏。

## 技术标书场景简要说明

选择「技术标书」后，将进入完整标书流程（大文本编辑模式默认）：

- 创建并维护 `plan.md`（技术方案简述、需求清单、指标性内容、撰写进度）。
- Phase 1–6：技术方案明确 → 需求确认 → 指标收集 → 计划与任务选择 → 迭代执行（小标题分解、零分点撰写、每约 1000 字暂停）→ 循环直至完成。
- 每个功能点包含：功能描述、实现流程、业务流程设计，并对应招标需求；正文采用公文层级，禁用分点符号（表格等例外）。

详细步骤与约束见 `references/technical-proposal.md`。

## 通用写作要求摘要

- **禁词**：避免「总之」「首先/其次/最后」「我们致力于」「这是一个…」「可以/能够」等，改用「综上所述」「第一…在此基础上…」「本方案旨在」「该模块承担…」「具备…能力」等。
- **去列表化**：正文以叙述性段落为主；表格、接口/参数说明等处可用列表或表格。
- **逻辑连接**：段落间用「基于上述…」「在…前提下」「针对…场景」等衔接。
- **输出控制**：每次输出完整小节；长文档每约 1000 字暂停确认；生成到文件时增量追加。

## 注意事项

1. 先明确「写哪类文档」，再进入对应场景或常规兜底。
2. 撰写前尽量确认使用场景与身份，便于语气与侧重点更贴合。
3. 行业用语为可选润色手段，按场景适当使用，不必处处加入。
4. 长文档按各场景要求分段确认（如标书每约 1000 字）；技术标书会维护 `plan.md`，请勿随意删除。
